After spending two weeks evaluating Harness from a testing perspective, I have concluded that the recent version of the Harness Cloud tool does not fully support testing for the banking domain.
Here are the key areas where the current version falls short:

Limited Testing Support: At a primary level, Harness is compatible mainly for build deployments and basic unit testing in general domains.
Lack of Advanced Testing Features: Features required for domain-specific testing, especially in banking, are missing or underdeveloped.
Insufficient Learning Resources: There is a noticeable lack of materials or videos or in devops community on social platforms to guide users on testing capabilities in Harness.

Currently, the tool seems to focus primarily on deployment automation, while testing support is still in its early stages within the industry.

Java Compiler Requirement (javac):
To work effectively in the Fiserv environment, we need javac in addition to java. This is essential for compiling flexible libraries, especially when installing plugins and reloading them. Currently, due to restrictions, Maven attempts to reach the central repository but fails because it requires a TLS certificate or proxy setup for this org.


JDK Version Compatibility in Harness:
Harness currently supports JDK 11 by default, but does not support JDK 17, 21, or 25. Our Fiserv laptops are configured with JDK 25, which creates a mismatch. As a result, we cannot test the code locally using commands like mvn test or mvn clean install before running the pipeline. Even if we add a JDK 25 installation step in Harness, it still defaults to JDK 11.


Testing and Domain Restrictions:
If we proceed with the default Harness setup without local testing, the pipeline execution takes significantly longer.
